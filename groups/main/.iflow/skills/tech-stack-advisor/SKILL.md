# Skill: Tech Stack Advisor - 技术栈选型顾问

## 描述
基于项目需求、团队技能、生态成熟度等因素，提供客观的技术栈选型建议。涵盖前端、后端、数据库、基础设施等各个层面。

## 触发条件
- 启动新项目需要选型
- 技术栈升级或迁移
- 评估新技术引入
- 解决现有技术栈痛点

## 选型维度

### 1. 项目需求分析

#### 功能需求
```
□ 应用类型：Web / Mobile / Desktop / API / CLI
□ 实时性要求：实时 / 近实时 / 离线
□ 数据复杂度：简单 CRUD / 复杂关系 / 大数据分析
□ 用户规模：内部工具 / 中小规模 / 大规模分布式
□ 多端需求：Web + Mobile + Desktop
```

#### 非功能需求
```
□ 性能要求：延迟敏感 / 吞吐量优先
□ 可用性要求：容忍停机 / 高可用（99.9%+）
□ 安全要求：标准 / 金融级 / 政府合规
□ 扩展性：垂直扩展足够 / 需要水平扩展
```

#### 风险评估清单
```
□ 社区活跃度：GitHub stars、贡献者数量
□ 维护状态：最近更新时间、issue 响应速度
□ 企业支持：是否有商业公司 backing
□ 人才市场：招聘难度、学习资源
□ 迁移成本：从现有技术迁移的难度
□ 供应商锁定：是否容易切换
```

## 常见场景推荐

### 场景 1：SaaS MVP（最小可行产品）
**目标**：快速验证想法，2-3 个月上线

**推荐栈**：
```
前端：Next.js + Tailwind CSS + shadcn/ui
后端：Next.js API Routes 或 FastAPI
数据库：PostgreSQL (Vercel/Supabase)
部署：Vercel
认证：Clerk 或 NextAuth
支付：Stripe
```

**理由**：
- 开发效率高，一人全栈可行
- 生态成熟，问题容易解决
- 扩展路径清晰

### 场景 2：高并发 API 服务
**目标**：处理 10K+ QPS，低延迟

**推荐栈**：
```
语言：Go 或 Rust
框架：Gin (Go) / Axum (Rust)
数据库：PostgreSQL + Redis 缓存
消息队列：NATS 或 Redis Streams
部署：Kubernetes
监控：Prometheus + Grafana
```

**理由**：
- 编译型语言性能优秀
- 资源占用低，成本可控
- 并发模型成熟

### 场景 3：AI/ML 应用
**目标**：集成 LLM、数据处理、模型服务

**推荐栈**：
```
语言：Python
框架：FastAPI 或 LangChain
LLM：OpenAI / Anthropic / 本地模型
向量数据库：Pinecone / Chroma / pgvector
工作流：Prefect / Airflow
部署：Docker + 云服务
```

**理由**：
- Python ML 生态最丰富
- 快速集成各种 AI 服务
- 数据处理工具链完善

### 场景 4：企业级内部系统
**目标**：稳定、可维护、团队可接手

**推荐栈**：
```
前端：React + Ant Design / Material-UI
后端：Java Spring Boot 或 .NET
数据库：PostgreSQL 或 SQL Server
部署：私有云或混合云
监控：ELK Stack + APM
```

**理由**：
- 人才储备充足
- 企业级支持和文档
- 长期维护有保障

## 选型流程

### 步骤 1：明确约束
```
硬性约束：
- 团队现有技能
- 预算限制
- 合规要求
- 遗留系统兼容

软性约束：
- 上线时间
- 性能目标
- 扩展预期
```

### 步骤 2：候选方案
```
列出 2-3 个可行方案，避免分析瘫痪
每个方案包含：
- 技术组合
- 优缺点
- 适用场景
- 风险点
```

### 步骤 3：POC 验证
```
对关键不确定性进行概念验证：
- 性能测试
- 开发效率评估
- 团队学习曲线
- 集成复杂度
```

## 反模式警示

### ❌ 避免过度工程
- 不要为可能永远不会到来的规模设计
- 先解决当前问题，再考虑未来扩展

### ❌ 避免追逐新潮
- 生产环境优先选择成熟技术
- 新技术先在 side project 试验

### ❌ 避免技术宗教
- 没有银弹，只有权衡
- 根据具体场景选择，而非个人偏好

### ❌ 忽视团队因素
- 考虑团队学习成本
- 考虑招聘难度
- 考虑长期维护
