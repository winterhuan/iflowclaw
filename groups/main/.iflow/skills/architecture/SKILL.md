# Skill: Architecture - 系统架构设计

## 描述
帮助设计可扩展、可维护的系统架构。从需求分析到技术选型，从数据流设计到部署架构，提供全面的架构指导。

## 触发条件
- 设计新系统或重构现有系统
- 技术选型决策
- 性能扩展规划
- 微服务拆分
- 数据库设计

## 设计原则

### 1. SOLID 原则
- **S**ingle Responsibility：单一职责
- **O**pen/Closed：开闭原则
- **L**iskov Substitution：里氏替换
- **I**nterface Segregation：接口隔离
- **D**ependency Inversion：依赖倒置

### 2. 架构风格选择

| 风格 | 适用场景 | 优点 | 挑战 |
|------|----------|------|------|
| 单体架构 | 小型应用、MVP | 简单、易部署 | 扩展困难 |
| 微服务 | 大型团队、复杂业务 | 独立部署、技术异构 | 运维复杂 |
| Serverless | 事件驱动、间歇负载 | 自动扩展、按量付费 | 冷启动、供应商锁定 |
| 事件驱动 | 异步处理、流式数据 | 松耦合、高吞吐 | 最终一致性 |
| 分层架构 | 企业应用 | 清晰分离、易测试 | 可能过度抽象 |

### 3. 质量属性权衡
```
        一致性
           ▲
           │
  可用性 ◄─┼─► 分区容错性
           │
           ▼
        性能

CAP 定理：三者不可兼得，根据业务优先级选择
```

## 设计流程

### 阶段 1：需求分析
1. **功能需求**：系统需要做什么？
2. **非功能需求**：
   - 性能：QPS、延迟要求
   - 可用性：SLA 目标（99.9%？99.99%？）
   - 扩展性：未来 6-12 个月的增长预期
   - 安全：合规要求（GDPR、SOC2 等）
3. **约束条件**：预算、团队技能、时间线

### 阶段 2：高层设计
使用 C4 模型进行分层设计：

```
┌─────────────────────────────────────────┐
│  Level 1: System Context                │
│  系统与外部实体的交互                    │
├─────────────────────────────────────────┤
│  Level 2: Container                     │
│  应用/服务/数据存储                       │
├─────────────────────────────────────────┤
│  Level 3: Component                     │
│  代码级别的组件组织                       │
├─────────────────────────────────────────┤
│  Level 4: Code                          │
│  类/接口级别的设计（可选）                │
└─────────────────────────────────────────┘
```

### 阶段 3：数据设计
1. **数据模型**：
   - 实体关系图（ERD）
   - 读写比例分析
   - 数据增长预估

2. **存储选型**：
   ```
   关系型数据库（PostgreSQL/MySQL）：事务型、复杂查询
   文档数据库（MongoDB）：灵活 schema、快速迭代
   缓存（Redis）：高频读取、临时数据
   搜索引擎（Elasticsearch）：全文搜索、日志分析
   时序数据库（InfluxDB/TimescaleDB）：指标、监控数据
   对象存储（S3）：文件、备份、静态资源
   ```

3. **数据流设计**：
   - 数据入口和出口
   - 转换和清洗流程
   - 备份和恢复策略

### 阶段 4：接口设计
1. **API 设计原则**：
   - RESTful 或 GraphQL？
   - 版本控制策略
   - 认证和授权机制
   - 限流和熔断策略

2. **事件设计**（如果使用事件驱动）：
   - 事件 schema
   - 消息队列选型
   - 消费者组设计

### 阶段 5：部署架构
```
┌─────────────────────────────────────────┐
│              CDN / WAF                  │
├─────────────────────────────────────────┤
│           Load Balancer                 │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │  App 1  │  │  App 2  │  │  App N  │ │
│  └─────────┘  └─────────┘  └─────────┘ │
├─────────────────────────────────────────┤
│           Cache Layer                   │
├─────────────────────────────────────────┤
│         Database Cluster                │
├─────────────────────────────────────────┤
│      Message Queue / Event Bus          │
└─────────────────────────────────────────┘
```

## 常见模式

### 微服务拆分策略
1. **按业务能力**：订单服务、用户服务、库存服务
2. **按子域**：核心域、支撑域、通用域
3. **按团队边界**：康威定律

### 扩展模式
- **水平扩展**：增加实例数量
- **垂直扩展**：提升单机配置
- **读写分离**：主从复制
- **分片**：数据水平拆分
- **缓存**：多级缓存策略

### 可靠性模式
- **熔断器**：防止级联故障
- **重试**：指数退避策略
- **限流**：令牌桶/漏桶算法
- **降级**：优雅降级策略
- **超时**：避免资源阻塞

### 架构文档结构
1. **概述**：系统目标和范围
2. **架构图**：C4 模型各层级
3. **技术栈**：语言、框架、工具
4. **数据流**：请求生命周期
5. **安全设计**：威胁模型和防护措施
6. **部署架构**：基础设施和运维
7. **决策记录**：关键决策和理由
