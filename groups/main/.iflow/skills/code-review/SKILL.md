# Skill: Code Review - 代码审查专家

## 描述
进行全面的代码审查，发现潜在问题、安全漏洞、性能瓶颈，并提供改进建议。遵循行业最佳实践和知名团队（Google、Microsoft、Anthropic）的代码审查标准。

## 触发条件
- 用户要求审查代码
- PR/MR 需要审查
- 代码重构前评估
- 学习代码审查技巧

## 审查维度

### 1. 功能性审查
```
□ 代码是否实现了预期功能？
□ 边界条件是否处理？
□ 错误处理是否完善？
□ 是否有潜在的逻辑错误？
```

### 2. 安全性审查
```
□ 输入验证是否充分？
□ 是否存在 SQL 注入/XSS/CSRF 风险？
□ 敏感数据是否妥善处理？
□ 权限检查是否正确？
□ 依赖项是否有已知漏洞？
```

### 3. 性能审查
```
□ 时间复杂度是否合理？
□ 是否存在 N+1 查询问题？
□ 内存使用是否高效？
□ 是否有不必要的计算？
□ 缓存策略是否恰当？
```

### 4. 可维护性审查
```
□ 命名是否清晰有意义？
□ 函数是否单一职责？
□ 代码是否易于测试？
□ 注释是否必要且准确？
□ 是否遵循项目规范？
```

### 5. 架构审查
```
□ 是否符合 SOLID 原则？
□ 耦合度是否过高？
□ 抽象层级是否恰当？
□ 是否引入不必要的复杂度？
```

## 输出格式

### 摘要
- **风险等级**：🔴 高风险 / 🟡 中风险 / 🟢 低风险
- **主要问题数**：X 个
- **建议改进数**：X 个
- **正面评价**：X 个

### 详细反馈
按严重程度排序，每个问题包含：
1. **位置**：文件路径和行号
2. **问题类型**：安全/性能/可维护性/功能
3. **描述**：具体问题
4. **影响**：可能导致的后果
5. **建议**：如何修复
6. **示例代码**：修复前后的对比

### 正面反馈
列出代码中的亮点：
- 设计模式使用得当
- 测试覆盖良好
- 文档清晰
- 性能优化巧妙

## 审查风格

### 建设性反馈示例
❌ **避免**："这段代码很糟糕"
✅ **推荐**："这里的循环嵌套可能导致 O(n²) 复杂度，建议考虑使用 Map 来优化到 O(n)"

❌ **避免**："你没处理错误"
✅ **推荐**："建议在第 45 行添加 try-catch，处理网络超时的情况，避免未捕获的异常导致服务崩溃"

## 安全检查清单（基于 OWASP）

### Web 应用
- [ ] 所有用户输入都经过验证和转义
- [ ] 身份验证和授权检查到位
- [ ] 敏感数据加密存储
- [ ] 使用参数化查询防止 SQL 注入
- [ ] 设置适当的 CSP 头
- [ ] 防范 CSRF 攻击

### API 设计
- [ ] 速率限制已实施
- [ ] 认证令牌安全传输
- [ ] 错误信息不泄露敏感信息
- [ ] 版本控制策略明确

## 语言特定检查点

### TypeScript/JavaScript
- 类型定义是否完整？
- 是否避免了 any 类型？
- 异步错误处理是否正确？
- 内存泄漏风险（事件监听器、闭包）？

### Python
- 类型提示是否使用？
- 是否遵循 PEP 8？
- 异常处理是否具体？
- 生成器是否合理使用？

### Go
- 错误处理是否完整？
- 是否有 goroutine 泄漏风险？
- 接口设计是否合理？
- 性能关键路径是否优化？
