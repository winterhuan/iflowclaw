# Skill: MVP Planner - MVP 规划专家

## 描述
帮助团队规划和执行最小可行产品（MVP），快速验证商业假设。从需求筛选到发布策略，提供系统化的 MVP 开发指南。

## 触发条件
- 启动新项目
- 验证产品想法
- 资源受限情况下的产品决策
- 寻找产品市场契合点（PMF）

## MVP 核心原则

### 1. 最小且可行
```
最小（Minimum）：
- 只做最核心的功能
- 砍掉"锦上添花"的特性
- 手动处理可以自动化的流程

可行（Viable）：
- 能解决用户的核心问题
- 用户体验足够好（不粗糙）
- 能收集到有效反馈
```

### 2. 构建-测量-学习循环
```
    ┌──────────┐
    │  构建     │
    │  (Build)  │
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │  测量     │
    │ (Measure) │
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │  学习     │
    │ (Learn)   │
    └────┬─────┘
         │
         └──────> 调整方向，再次构建
```

## MVP 规划流程

### 阶段 1：问题验证

#### 问题陈述模板
```
我们的目标用户是 [谁]，
他们面临 [什么问题]，
目前的解决方案是 [现有方案]，
但存在的问题是 [痛点]，
所以我们想提供 [新方案]。
```

#### 风险假设识别
```
□ 需求风险：用户真的有这个问题吗？
□ 方案风险：我们的方案能解决问题吗？
□ 商业风险：用户愿意为此付费吗？
□ 技术风险：技术上可行吗？
□ 市场风险：市场规模足够大吗？
```

### 阶段 2：功能筛选

#### MoSCoW 方法
```
Must have（必须有）：
- 没有它产品就没价值
- 核心功能，不可妥协
- 通常占 20% 功能，提供 80% 价值

Should have（应该有）：
- 重要但不是立即需要
- 可以手动 workaround
- 下一个迭代加入

Could have（可以有）：
- 锦上添花的功能
- 有更好，没有也行
- 根据资源决定

Won't have（暂不做）：
- 明确排除在当前范围外
- 避免范围蔓延
- 记录到 backlog
```

#### 功能优先级矩阵
```
         用户价值高
              ▲
              │
    ┌─────────┼─────────┐
    │  快速   │  核心   │
    │  获胜   │  功能   │
    │  (先做) │  (必须) │
实现├─────────┼─────────┤ 成本
成本│  低价值 │  填充   │
低  │  (不做) │  (后做) │ 高
    └─────────┼─────────┘
              │
              ▼
         用户价值低
```

### 阶段 3：MVP 范围定义

#### 用户故事地图
```
用户活动：浏览商品 -> 添加到购物车 -> 结账 -> 收货

MVP 范围（第 1 列）：
├─ 浏览商品：查看商品列表、查看商品详情
├─ 添加到购物车：添加商品、修改数量
├─ 结账：填写地址、选择支付方式、确认订单
└─ 收货：查看订单状态

后续迭代：
├─ 搜索过滤、推荐系统
├─ 收藏、优惠券
├─ 多种支付方式、分期付款
└─ 物流追踪、评价系统
```

#### 时间盒约束
```
理想 MVP 时间：2-4 周
最长不超过：6-8 周

如果超出，继续削减范围，直到能 fit 进时间盒
```

### 阶段 4：技术策略

#### 技术选型原则
```
速度 > 完美
- 选择团队最熟悉的技术
- 使用现成的解决方案（SaaS、开源）
- 避免过早优化

可扩展性考虑
- 选择有清晰扩展路径的技术
- 数据模型预留扩展空间
- 但不要为"可能"的规模过度设计
```

#### 推荐技术栈（快速启动）
```
前端：
- Next.js + Vercel（快速部署）
- Tailwind CSS（快速样式）
- shadcn/ui（现成组件）

后端：
- Supabase / Firebase（BaaS，快速启动）
- 或 Next.js API Routes（全栈 JS）

数据库：
- PostgreSQL（Supabase 提供）
- 或 Firebase Firestore（文档型）

认证：
- Clerk / Auth0（快速集成）
- 或 NextAuth（自托管）

支付：
- Stripe（标准选择）

文件存储：
- AWS S3 / Cloudflare R2
- 或 Supabase Storage
```

### 阶段 5：发布策略

#### 发布阶段
```
1. 内部测试（Dogfood）
   - 团队内部使用
   - 发现明显问题

2. 封闭测试（Alpha）
   - 邀请制，小范围用户
   - 收集深度反馈
   - 10-50 个用户

3. 公开测试（Beta）
   - 公开注册，但标注"测试版"
   - 扩大用户群
   - 100-1000 个用户

4. 正式发布（GA）
   - 去除测试标识
   - 全面推广
```

#### 早期用户获取
```
□ 个人网络：朋友、同事、社交媒体
□ 垂直社区：Reddit、Discord、微信群
□ 内容营销：博客、视频、播客
□ 冷启动：一对一邀请目标用户
□ 黑客新闻：Product Hunt、Hacker News
```

## MVP 成功指标

### 海盗指标（AARRR）
```
Acquisition（获取）：
- 注册用户数
- 获客成本（CAC）

Activation（激活）：
- 完成核心行为的用户比例
- 例如：完成首次购买、创建首个项目

Retention（留存）：
- 次日/周/月留存率
-  cohort 分析

Revenue（收入）：
- 付费转化率
- 客单价（ARPU）
- 生命周期价值（LTV）

Referral（推荐）：
- 病毒系数（K-factor）
- 净推荐值（NPS）
```

### 关键指标选择
```
□ 虚荣指标（避免）：
  - 总下载量
  - 总注册数
  - 页面浏览量

□ 可操作指标（推荐）：
  - 活跃用户比例
  - 功能使用率
  - 用户满意度
  - 付费转化率
```

## 常见陷阱

### ❌ 避免
- **完美主义**：花 6 个月做"完整版"MVP
- **功能蔓延**：不断添加"就这一个功能"
- **技术过度工程**：微服务、K8s 过早引入
- **忽视反馈**：闭门造车，不与用户交流
- **过早规模化**：还没验证就大规模推广

### ✅ 推荐
- **快速发布**：2-4 周上线第一个版本
- **无情砍功能**：只做最核心的
- **手动处理**：能手动就不自动化
- **持续访谈**：每周至少 5 个用户访谈
- **数据驱动**：定义指标，持续测量

## MVP 检查清单

### 发布前
```
□ 核心功能完成且可用
□ 关键流程无阻断性 bug
□ 有反馈收集渠道
□ 基础分析埋点
□ 法律合规（隐私政策、ToS）
```

### 发布后（第 1 周）
```
□ 监控错误和性能
□ 收集首批用户反馈
□ 修复关键问题
□ 与用户一对一访谈
```

### 发布后（第 1 月）
```
□ 分析使用数据
□ 识别流失原因
□ 规划下一个迭代
□ 评估 PMF 信号
```
